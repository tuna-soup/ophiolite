# Processing Cache Architecture And Benchmarking

## Audience and intent

This note documents the processing-cache experiment for TraceBoost trace-local volume processing.

It covers:

- the original problem
- the candidate cache designs
- the benchmark method
- the measured results
- the final architecture decision

This is an engineering writeup, not a product pitch. The point is to record why some cache paths were kept and why others were removed after measurement.

## Problem statement

TraceBoost lets users build multi-step trace-local processing pipelines and run them either as preview on the current section or as full-volume materialization.

The performance concern is straightforward:

- a user runs a `5` to `10+` step pipeline on a `tbvol`
- the user changes one later parameter
- the current naive behavior reruns the entire pipeline from step `1`

That is attractive from a simplicity standpoint, but it becomes increasingly wasteful as datasets and pipelines grow.

The question was:

> can we avoid rerunning unchanged prefixes without creating an unacceptable amount of storage overhead and code complexity?

## Baseline architecture before the cache work

Before this experiment, the behavior was simple:

- every full-volume processing run materialized a fresh derived store
- there was no exact-rerun reuse layer
- there was no reusable prefix cache
- a late edit still reran the full pipeline

The important positive property of the baseline was clarity:

- one request
- one full materialization
- one visible derived output

The downside was obvious recompute.

## Candidate cache architecture that was prototyped

The experiment added three distinct reuse mechanisms.

### 1. Exact full-rerun reuse

If the source store fingerprint and full pipeline hash match an existing visible derived output, return that existing result instead of recomputing.

This is the cheapest form of reuse:

- no hidden intermediate storage is required
- lookup is content-based
- the returned artifact is already a user-visible result

### 2. Explicit checkpoint reuse

If the user inserts explicit pipeline checkpoints, TraceBoost can reuse the latest matching checkpoint store and rerun only the suffix.

This is still a user-directed tradeoff:

- the user is opting into visible intermediate artifacts
- storage growth is explicit rather than hidden
- reuse is bounded by the checkpoints the user chose

### 3. Automatic hidden prefix checkpoints

The experimental `Balanced` path added one hidden checkpoint for the longest prefix before the final operator, stored under the hidden processing cache rather than the normal derived-output inventory.

This was the risky part of the design:

- more hidden storage
- more planner logic
- more cache invalidation logic
- more code complexity

It only made sense to keep if the measured rerun savings were strong.

## Additional runtime issue found during benchmarking

The benchmark also exposed a separate correctness issue in the processing path.

The current runtime materializer requires reader and writer tile geometry to match exactly. During benchmarking, runs against `f3.tbvol` failed unless the output materialization reused the input store's tile shape explicitly.

That led to a backend fix in both:

- `app/traceboost-frontend/src-tauri`
- `app/traceboost-app`

The final processing path now derives `MaterializeOptions.chunk_shape` from the input store rather than relying on default chunk selection.

This was not the original cache goal, but it was a real backend bug surfaced by the cache benchmark work.

## Benchmark method

The benchmark lives in:

- `app/traceboost-frontend/src-tauri/src/processing_cache_bench.rs`

It was run in release mode with:

```powershell
cargo test -p traceboost-desktop benchmark_processing_cache_f3 --release -- --ignored --nocapture
```

The harness intentionally measures:

- uncached full rerun
- cached first run
- exact-output hit
- late-edit rerun
- middle-edit rerun

The rerun cases were corrected so that setup and cache priming happen outside the timed portion. That matters because otherwise the benchmark would mostly time the initial run instead of the actual reuse path.

## Datasets used

The benchmark used three datasets:

### 1. Real small control

- `test-data/f3.tbvol`
- shape: `23 x 18 x 75`

### 2. Larger F3-derived dataset

- `f3-upscaled-level-3.tbvol`
- generated by applying `upscale_store` three times to `f3.tbvol`
- shape: `177 x 137 x 75`

This second dataset is not a synthetic cube in the same sense as the storage benchmark document. It is a deterministic F3-derived larger volume used as a practical proxy to make rerun costs visible without introducing a separate synthetic data generator into the frontend benchmark first.

### 3. Real larger F3 import

- `f3_dataset-smoke.tbvol`
- ingested from `C:\Users\crooijmanss\Downloads\archive\f3_dataset.sgy`
- shape: `651 x 951 x 462`
- tile shape: `41 x 56 x 462`

This is the first benchmark dataset in this note that is both materially large and directly derived from the larger SEG-Y the user wanted to test.

## Pipeline matrix used

The benchmark used representative synthetic pipelines:

- `5-step` late-edit case
- `8-step` late-edit case
- `10-step` middle-edit case

The operator mix included:

- amplitude scalar
- AGC RMS
- phase rotation
- lowpass/highpass/bandpass filters

## Acceptance bar

The benchmark plan set this bar for keeping the automatic cache path:

- at least `25%` wall-clock improvement on representative late-edit reruns
- no more than `10%` regression on first-run cache-miss cases

If the automatic cache path failed that bar, it was supposed to be removed or heavily simplified.

## Interpretation

### Exact full-rerun reuse is worth keeping

This is the cleanest signal in the benchmark.

On the larger F3-derived dataset:

- exact lookup: `2.984 ms`
- uncached `5-step` full rerun: `45.091 ms`

That is roughly a `93%` reduction in wall-clock time for exact reruns, with no hidden intermediate materialization required.

On the real larger imported F3 dataset:

- exact lookup: `37.574 ms`
- uncached `5-step` full rerun: `14069.217 ms`

That is roughly a `99.7%` reduction in wall-clock time for exact reruns.

This part of the cache design is justified.

### Automatic hidden prefix checkpoints did not clear the bar

The experimental automatic prefix path failed the acceptance criteria.

On the larger dataset:

- first-run overhead regressed from `45.091 ms` to `110.448 ms`
- the `5-step` late-edit rerun was effectively flat: `44.724 ms` versus `45.091 ms`
- the `8-step` late-edit rerun was slightly worse than baseline: `45.489 ms` versus `41.738 ms`
- the `10-step` middle-edit rerun was much worse: `83.772 ms` versus `46.717 ms`

This misses both bars:

- the late-edit speedup is nowhere near `25%`
- the first-run regression is far worse than `10%`

The conclusion is simple: automatic hidden prefix checkpointing is not justified in the current implementation.

The real larger imported F3 dataset confirms the same conclusion:

- first-run overhead regressed from `14069.217 ms` to `58615.459 ms`
- the `5-step` late-edit rerun improved only slightly: `13323.668 ms` versus `14069.217 ms`
- the `8-step` late-edit rerun improved, but still only to `12165.425 ms` from `13993.871 ms`
- the `10-step` middle-edit rerun was dramatically worse: `60461.283 ms` versus `13246.733 ms`

This is still nowhere near the acceptance bar for keeping the automatic hidden-checkpoint path.

### Explicit checkpoint reuse is less bad, but not a strong enough default

The explicit checkpoint case performed somewhat better than the automatic hidden-prefix case, but still not well enough to justify making that strategy implicit.

On the larger dataset:

- uncached `5-step` rerun: `45.091 ms`
- explicit checkpoint late edit rerun: `40.664 ms`

That is only about a `10%` improvement. It is directionally better, but not a compelling default policy.

On the real imported F3 dataset:

- uncached `5-step` rerun: `14069.217 ms`
- explicit checkpoint late edit rerun: `12840.818 ms`

That is only about an `8.7%` improvement.

Still, explicit checkpoints remain more defensible because:

- the user explicitly asked for them
- the intermediate artifacts are visible rather than hidden
- the storage cost is intentional

### Small datasets make prefix caching look actively bad

On `f3.tbvol`, all prefix-cache variants were clearly worse than just rerunning.

That is not surprising:

- the dataset is tiny
- the full rerun is already very fast
- cache planning and checkpoint management dominate the cost

This reinforces the broader point that cache complexity cannot be justified by intuition alone.

## Final architecture decision

The benchmark changed the design.

### Kept

- exact full-rerun reuse
- cache/index infrastructure needed to support exact reuse
- explicit checkpoint reuse
- benchmark harness for future reevaluation

### Removed from the production path

- automatic hidden prefix checkpoint injection for `Balanced`

The code was simplified to match that decision. Production processing no longer auto-inserts a hidden prefix stage.

## Final architecture after benchmarking

The resulting architecture is:

1. A full-volume run still produces a visible derived output.
2. If the exact same request is rerun later, TraceBoost can return the existing derived output immediately.
3. If the user explicitly created checkpoints, later compatible reruns can reuse those checkpoints.
4. TraceBoost does not automatically create hidden prefix checkpoints in the normal production path.

The live cache implementation has since been simplified to match that narrower decision:

- a single enabled/disabled reuse switch rather than multiple cache policy modes
- visible output reuse and visible checkpoint reuse only in the production path
- no production hidden-prefix artifact management surface

That is a narrower design than the original cache prototype, but it is the design supported by the measurements.

## Decision points and lessons

### 1. Exact-result reuse and prefix reuse are not the same bet

They should not be evaluated as one feature.

- exact-result reuse is cheap and effective
- automatic prefix reuse is materially more complex and did not pay for itself here

### 2. Hidden cache state needs a higher performance bar than visible user artifacts

If the user can see and intentionally manage checkpoints, a moderate win may still be acceptable.

If the system is going to create hidden large artifacts automatically, the measured win needs to be strong. It was not.

### 3. Benchmarking uncovered a runtime contract issue

The tile-geometry mismatch requirement was real and needed to be fixed regardless of the cache outcome.

That alone justifies having built and run the benchmark harness.

### 4. Small datasets are dangerous for cache decisions

Tiny control cubes are good for smoke tests, but they are not enough to justify storage-heavy execution features.

That is why the benchmark was extended beyond raw `f3.tbvol`.

## What remains open

This does not prove that prefix reuse is never worthwhile. It proves that the current automatic implementation is not worthwhile on the tested workloads.

The question can be reopened later if all of the following are true:

- larger real customer datasets are benchmarked
- more expensive operator mixes are benchmarked
- prefix checkpoint creation becomes cheaper
- rerun planning becomes demonstrably better than the current one-prefix strategy

Until then, the simpler architecture is the better one.

## Final conclusion

The processing-cache experiment produced one strong keeper and one clear rollback.

- keep exact full-rerun reuse
- do not keep automatic hidden prefix checkpoints in the production path

That is the right outcome.

The benchmark did what it was supposed to do:

- it prevented a speculative cache design from becoming permanent without evidence
- it preserved the part that clearly helps
- it forced a simplification where the numbers did not justify the overhead
